<objective>
Wire the session module into both workspace and bundle execution so remote targets use persistent named tmux sessions.

Purpose: This is the core behavior change for Phase 2. When a workspace has `host` set, panout sends `ssh -t host "tmux new-session -A -s <session_name>"` to the first pane instead of bare SSH to every pane. When a bundle references server configs with SSH hosts, panout routes through the same session mechanism using the bundle name as the session name. This creates persistent remote tmux sessions that survive disconnects (SESS-01, SESS-02, SESS-05). Workspaces without `host` and bundles without server targets are completely unchanged (SESS-06).

Output: Modified `src/main.rs` with remote session routing in both `run_workspace()` and `run_bundle()`.
</objective>

    ```rust
    fn run_workspace(config: &Config, name: &str) -> Result<()> {
        let workspace = config
            .get_workspace(name)
            .ok_or_else(|| PanoutError::WorkspaceNotFound(name.into()))?;

        match &workspace.host {
            Some(host) => {
                // Remote session: SSH into host with named tmux session
                let cmd = session::build_remote_session_cmd(
                    host,
                    name,
                    workspace.dir.as_deref(),
                );
                let panes = tmux::pane_indices()?;
                tmux::send_keys(panes[0], &cmd)?;
            }
            None => {
                // Local workspace: create windows/panes as before
                let start_window = tmux::current_window()?;
                run_workspace_windows(workspace)?;
                tmux::select_window(start_window)?;
            }
        }

        Ok(())
    }
    ```

    Modify `run_bundle` in src/main.rs to detect when a bundle's resolved commands target SSH servers and route through the session module.

    The bundle execution path currently resolves commands via `resolver::resolve_with_panes` and sends them to local tmux panes. When the bundle's resolved commands include SSH connections to servers (from `[servers.*]` config entries), the bundle should instead create a named remote tmux session.

    Detection approach: Check if the config has server entries that are referenced by the bundle's resolved commands. A simpler approach: look at the resolved pane_commands -- if any command starts with `ssh `, the bundle targets a remote host.

    Implementation in `run_bundle`:
    1. After resolving `pane_commands`, scan the first pane's commands for SSH patterns. Specifically check if any command in pane 0 starts with `"ssh "` or matches server host patterns from `config.servers`.
    2. If remote SSH detected: extract the host from the SSH command (or look it up from config.servers), build the session command using `session::build_remote_session_cmd(host, bundle_name, None)`, and send it to the first pane. The bundle name (e.g., "dev.frontend") becomes the session name after sanitization.
    3. If no SSH commands detected: continue with the existing flow (create panes, send commands).

    Alternative simpler approach (preferred): Instead of inspecting resolved commands, add a helper that checks if a bundle's commands reference any `@servers.*` entries or if the config has an explicit server association. For MVP, the simplest approach is:
    - After creating panes and before sending commands, check if any resolved command for pane 0 matches `ssh <host>` where host is a known server host from `config.servers`.
    - If matched: replace that pane's commands with a single `session::build_remote_session_cmd(host, bundle_name, None)` call.
    - If not matched: send commands as before.

    ```rust
    fn run_bundle(cli: &Cli, config: &Config) -> Result<()> {
        let bundle_name = cli
            .bundle
            .as_ref()
            .ok_or_else(|| PanoutError::BundleNotFound("no bundle specified".into()))?;

        let num_panes = cli.num.unwrap_or(1);
        let pane_commands = resolver::resolve_with_panes(config, bundle_name)?;

        let bundle = config.get_bundle(bundle_name);
        let layout = cli
            .layout()
            .or_else(|| bundle.and_then(|b| b.layout))
            .or(config.defaults.layout)
            .unwrap_or(Layout::Tiled);

        let pane_indices = tmux::create_panes(num_panes, layout)?;

        // Check if pane 0 commands include an SSH connection to a known server
        let server_host = find_server_host(&pane_commands, config);

        if let Some(host) = server_host {
            // Remote bundle: create named tmux session on remote host
            let cmd = session::build_remote_session_cmd(
                &host,
                bundle_name,
                None,
            );
            if let Some(&pane) = pane_indices.first() {
                tmux::send_keys(pane, &cmd)?;
            }
        } else {
            // Local bundle: send commands to panes as before
            for (i, commands) in pane_commands {
                if let Some(&actual_pane) = pane_indices.get(i as usize) {
                    for cmd in commands {
                        tmux::send_keys(actual_pane, &cmd)?;
                    }
                }
            }
        }

        Ok(())
    }

    /// Check if resolved pane commands contain an SSH command targeting a known server.
    fn find_server_host(
        pane_commands: &[(u32, Vec<String>)],
        config: &Config,
    ) -> Option<String> {
        let known_hosts: Vec<&str> = config.servers.values()
            .map(|s| s.host.as_str())
            .collect();

        for (_, commands) in pane_commands {
            for cmd in commands {
                if let Some(rest) = cmd.strip_prefix("ssh ") {
                    let target = rest.trim();
                    if known_hosts.iter().any(|h| target == *h || target.ends_with(h)) {
                        return Some(target.to_string());
                    }
                }
            }
        }
        None
    }
    ```

    IMPORTANT constraints:
    - The `find_server_host` helper is a simple heuristic matching resolved SSH commands against known server hosts from config. This covers the standard case where bundles reference `@servers.*` entries.
    - Bundles that don't involve SSH servers continue unchanged (SESS-06).
    - The session name is the bundle name (e.g., "dev.frontend") which gets sanitized to "dev-frontend" by the session module.
