<objective>
Wire the session module into workspace execution so remote workspaces use persistent named tmux sessions.

Purpose: This is the core behavior change for Phase 2. When a workspace has `host` set, panout sends `ssh -t host "tmux new-session -A -s <session_name>"` to the first pane instead of bare SSH to every pane. This creates a persistent remote tmux session that survives disconnects (SESS-01, SESS-02, SESS-05). Workspaces without `host` are completely unchanged (SESS-06).

Output: Modified `src/main.rs` with remote session routing in `run_workspace_windows()`.
</objective>

    2. When workspace has `host`, instead of calling `run_workspace_windows`, build the remote session command using `session::build_remote_session_cmd(host, name, workspace.dir.as_deref())` and send it to the current pane via `tmux::send_keys`.     3. When workspace has no `host`, continue with the existing flow (call `run_workspace_windows` as before).

    The updated `run_workspace` function should look like:
    ```rust
    fn run_workspace(config: &Config, name: &str) -> Result<()> {
        let workspace = config
            .get_workspace(name)
            .ok_or_else(|| PanoutError::WorkspaceNotFound(name.into()))?;

        match &workspace.host {
            Some(host) => {
                // Remote session: SSH into host with named tmux session
                let cmd = session::build_remote_session_cmd(
                    host,
                    name,
                    workspace.dir.as_deref(),
                );
                let panes = tmux::pane_indices()?;
                tmux::send_keys(panes[0], &cmd)?;
            }
            None => {
                // Local workspace: create windows/panes as before
                let start_window = tmux::current_window()?;
                run_workspace_windows(workspace)?;
                tmux::select_window(start_window)?;
            }
        }

        Ok(())
    }
    ```

    IMPORTANT constraints:
    - Do NOT modify `run_workspace_windows` -- it stays exactly as-is for the local flow.
    - Do NOT touch `run_bundle` -- bundles are separate from workspace remote sessions.
    - The remote flow sends a single SSH command to the current pane's first index. The user lands in the remote tmux session. No local windows/panes are created for remote workspaces.
    - This approach satisfies SESS-01 (named session created), SESS-02 (reattach via -A flag), SESS-05 (reconnect after disconnect), and SESS-06 (local workspaces unchanged).
